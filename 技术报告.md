## 1. 项目背景

本项目是一套运行于 Rockchip RK356X 边缘计算平台的实时目标检测流水线，面向工业/安防场景，要求在资源受限的嵌入式设备上实现低延迟、高可靠的 AI 推理。

核心需求：
- 实时采集摄像头视频（V4L2），通过 NPU 加速 YOLO 目标检测
- 检测结果通过 MQTT 上报，同时提供 HTTP/MJPEG 网页实时预览
- 全流程端到端延迟 < 100ms，支持 720p 分辨率

技术选型：
- **NPU 推理**：RKNN Runtime（rknpu2），充分利用 RK356X 内置 NPU
- **图像预处理**：RGA 硬件加速（零拷贝色彩空间转换 + 缩放）
- **进程间通信**：POSIX 共享内存 + seqlock，避免推理进程与 Web 进程的数据竞争
- **消息总线**：MQTT（libmosquitto），解耦检测结果与业务告警

## 5. 视频格式与解码方案

### 5.1 采集格式

摄像头输出 **YUYV 4:2:2**（YUY2），每像素 2 字节，640×480 分辨率，原始帧大小约 **614,400 字节（600KB）**。

### 5.2 RGA 硬件预处理

RKNN 模型输入要求 RGB888 640×640，需要：
1. 色彩空间转换：YUYV → RGB888
2. 分辨率缩放：640×480 → 640×640（letterbox 填充）

使用 **RGA（Raster Graphic Acceleration）** 硬件单元完成，避免 CPU 软件转换。

关键 API：
```cpp
// 正确格式（修复后）
src_img.format = RK_FORMAT_YUYV_422;
dst_img.format = RK_FORMAT_RGB_888;
improcess(src_img, dst_img, pat_img, src_rect, dst_rect, dst_rect, IM_ALPHA_BLEND_DST_OVER);
```

**修复前的错误**：使用 `RK_FORMAT_YCbCr_422_P`（平面格式）处理 packed YUYV 数据，导致色彩通道错乱，推理结果全部错误。

### 5.3 RKNN 模型输入

| 参数 | 值 |
|------|----|
| 格式 | RGB888 |
| 尺寸 | 640×640 |
| 数据类型 | uint8 |
| 内存类型 | `RKNN_TENSOR_VIRTUAL_ADDRESS`（mmap 零拷贝） |

## 6. RKNN 部署流程

```
1. 模型转换（PC 端）
   PyTorch/ONNX → rknn-toolkit2 → model.rknn

2. 模型加载（设备端）
   rknn_init(&ctx, model_data, model_size, 0, NULL)

3. 输入配置
   rknn_set_io_mem(ctx, input_mem, &input_attr)
   // input_attr.type = RKNN_TENSOR_UINT8
   // input_attr.fmt  = RKNN_TENSOR_NHWC

4. 推理执行
   rknn_run(ctx, NULL)  // 同步，阻塞至 NPU 完成

5. 输出读取
   rknn_get_output(ctx, n_output, outputs, 1)
   // 按 grid area 降序排序，确保 stride 顺序正确

6. 后处理
   post_process(outputs, BOX_THRESH, NMS_THRESH, &detect_result_group)

7. 资源释放
   rknn_destroy(ctx)
```

**输出张量排序逻辑**（修复关键）：

RKNN 返回的输出张量顺序不保证与 stride 对应，需按 `grid_h × grid_w`（即 grid area）从大到小排序，确保传入 `post_process` 的顺序为 stride 8 → stride 16 → stride 32。

## 9. 问题排查记录

### 问题 1：推理结果全部错误，检测框坐标异常

**现象**：模型加载成功，推理无报错，但输出检测框坐标全部错误或无检测结果。

**排查过程**：
1. 添加 `dump_tensor_attr` 打印输入输出张量属性，确认模型输入格式为 RGB888
2. 检查 RGA 转换代码，发现源格式设置为 `RK_FORMAT_YCbCr_422_P`（平面 YUV）
3. 实际摄像头输出为 YUYV packed 格式，两者内存布局完全不同

**根因**：RGA 源格式与实际数据格式不匹配，导致色彩通道错乱，模型输入数据损坏。

**修复**：将 RGA 源格式改为 `RK_FORMAT_YUYV_422`，API 从 `imresize` 改为 `improcess`。

**文件**：[src/inference.cpp](src/inference.cpp)

---

### 问题 2：检测框坐标系统性偏移

**现象**：检测框能检测到目标，但坐标相对实际位置有规律性偏移。

**排查过程**：
1. 打印 RKNN 输出张量的 `dims` 属性，发现 3 个输出张量的 grid size 分别为 80×80、40×40、20×20
2. 检查 `post_process` 函数，发现其假设输入顺序为 stride 8（80×80）→ stride 16（40×40）→ stride 32（20×20）
3. 实际 RKNN 返回顺序与预期不符

**根因**：RKNN 输出张量顺序不保证，`post_process` 收到错误顺序的特征图。

**修复**：实现 `get_output_grid_area` 函数，按 grid area 降序排序后再传入 `post_process`。

**文件**：[src/inference.cpp](src/inference.cpp)

---

### 问题 3：web_viewer 画面周期性卡顿

**现象**：MJPEG 流每隔数秒出现 1-2 帧画面冻结。

**排查过程**：
1. 分析共享内存写入时序：原代码在推理完成后才写入 YUYV 原始帧
2. 当推理耗时较长（如目标较多时 NMS 耗时增加），web_viewer 读到的是上一帧的旧数据

**修复**：调整写入顺序，每帧进入推理线程后**立即**写入 YUYV 原始帧和 `seq`，推理完成后再更新检测框。

**文件**：[src/inference.cpp](src/inference.cpp)

---

### 问题 4：共享内存 seqlock 写端未实现

**现象**：代码审查发现 `SharedFrame.lock_seq` 字段存在，`web_viewer` 读端已实现 seqlock，但 `inference.cpp` 写端未对 `lock_seq` 进行奇偶写保护。

**状态**：**未修复**，存在低概率读写竞态风险。

**修复方案**：
```cpp
// inference.cpp 写端应添加：
shm->lock_seq.fetch_add(1, std::memory_order_release); // 写前置奇
// ... 写入数据 ...
shm->lock_seq.fetch_add(1, std::memory_order_release); // 写后置偶
```

## 10. 后续改进方向

### 优先级排序

| 优先级 | 改进项 | 预期收益 | 难度 |
|--------|--------|----------|------|
| P0 | 完成 seqlock 写端实现 | 消除共享内存竞态 | 低 |
| P1 | RKNN 异步推理（`rknn_run` 非阻塞） | NPU 利用率 50% → 90%，FPS +30% | 中 |
| P1 | 暴露 `drop_count_` 到日志/指标 | 可观测性，便于调优队列深度 | 低 |
| P2 | 零拷贝采集（V4L2 DMA-BUF → RGA） | 消除 capture→inference 的 memcpy | 高 |
| P2 | MJPEG 多客户端共享编码 | 多客户端时 CPU 开销线性增长 → 恒定 | 中 |
| P3 | alarm 线程改为条件变量唤醒 | 消除 5ms 轮询，降低 CPU 占用 | 低 |
| P3 | 支持 H.264 硬件编码输出 | 降低 MJPEG 带宽占用 80% | 高 |

### 10.1 RKNN 异步推理（最高收益）

当前 `rknn_run` 为同步调用，NPU 运行期间 CPU 线程阻塞等待，NPU 利用率约 50%。

改进方案：使用 `rknn_run` 的异步模式 + `rknn_wait`，在 NPU 推理期间 CPU 可并行处理下一帧的 RGA 预处理，形成流水线：

```
帧N:  [RGA预处理] [NPU推理........] [后处理]
帧N+1:            [RGA预处理]       [NPU推理........] [后处理]
```

预期 FPS 从 28 提升至 38+。

### 10.2 零拷贝采集链路

当前数据路径存在 2 次大块内存拷贝：
1. V4L2 mmap → `RawFrame`（~600KB）
2. `RawFrame` → RGA 源缓冲区（~600KB）

通过 V4L2 DMA-BUF 导出 + RGA 直接消费，可消除两次拷贝，节省约 **36MB/s** 内存带宽（30fps 时）。

### 10.3 可观测性完善

建议添加以下指标输出（每 5 秒打印一次）：
- 各队列丢帧率（`cap_queue.drop_count_` / `infer_queue.drop_count_`）
- 各阶段平均耗时（采集、RGA、推理、后处理）
- MQTT 发布成功率
